What is an “agent harness”?

  • Deterministic code + engineered prompts around the model
  • The set of abstractions which transforms IO machine into “agent”

In Pi, the default tool belt is read, write, edit, and bash.

What is harness engineering?

Definition

  • Designing the system around the model
  • Shaping how the agent sees, acts, and fits your workflow
  • More than prompting: tools, UX, memory, constraints, extensions

Why Pi matters

  • Small core of strong defaults
  • Extensions as first-class primitive
  • Install or write the missing piece
  • Learn it all, as you go

Pi’s modes

Same harness, four different surfaces.

Interactive

TUI for humans

Editor, chat history, sessions, commands, model switching.

pi

Print / JSON

CLI for scripts

One-shot output, or stream every event as JSON lines.

pi -p "..."
pi --mode json "..."

SDK

Embed the agent

Use the same session engine directly from Node/TypeScript.

createAgentSession()

RPC

Process integration

Headless JSONL protocol over stdin/stdout for non-JS IDEs and other apps.

pi --mode rpc

Pi documents four run surfaces: interactive mode, print/JSON output, RPC mode, and the SDK. The point of this slide is that Pi is not just a terminal app; the same harness can be used by humans, scripts, subprocess integrations, or embedded directly in code.

Compose a Pi for one job

A custom session can be just a Bash alias.

alias researcher='pi \
  --tools read,bash,grep,find,ls \
  --extension ~/.pi/agent/extensions/web-search.ts \
  --extension ~/.pi/agent/extensions/fetch-content.ts \
  --no-session'
Read-only researcher

Same Pi, different affordances

  • No write or edit
  • Adds research extensions
  • Easy to launch, easy to share
This is one of Pi’s superpowers: you do not need to fork the product to make a different agent. Compose the session you want at launch time. Here the alias creates a read-only research specialist by changing the tool belt and loading extra extensions.

Aliases are jobs.* Presets are states.

A preset is a named bundle of model, thinking, tools, and instructions.

What they are

Switchable workflow shapes inside Pi

  • Start with --preset plan
  • Switch later with /preset
  • Define globally or in .pi/presets.json
What they’re for

Plan → implement → review

  • Aliases launch a role
  • Presets express a state transition
  • CLI flags still override when needed

*Or whatever you want them to be. You know the Shell.

Define what a preset is before selling it: a named bundle of model, thinking level, tools, and instructions. The key idea is that aliases launch a job, while presets let you move between workflow states like plan, implement, and review without leaving Pi.

Tiny example extension:
Claude-only prompt injection

export default function (pi) {
  let claudePrompt;

  pi.on("model_select", async ({ model }) => {
    claudePrompt = model.id.includes("claude")
      ? "Seek existing windows for this shootout before blasting new holes, cowboy."
      : null;
  });

  pi.on("before_agent_start", async (event) => {
    if (!claudePrompt) return;
    return { systemPrompt: `${event.systemPrompt}\n\n${claudePrompt}` };
  });
}
What the heck?

Prompt logic can follow the active model

  • model_select updates extension state
  • before_agent_start injects per-turn
  • No Claude, no Claude-specific system prompt
This is intentionally short. The extension watches model changes, stores a Claude-only instruction, and appends it only at turn start. Because the modification happens in before_agent_start, the extra prompt is not permanently baked in; when you switch away, future turns stop receiving it.

Engineer brain

Dependency injection for the harness

Model choice, tool belt, prompts, auth, and UI are not “the product.” They’re parameters you can bind differently for each repo, team, or workflow.

This is the close-the-sale slide. The point is not that Pi has many settings. The point is that harness configuration is itself composable software. Different repositories, tasks, or integrations can bind the same engine to different tools, prompts, auth, and UI.

Pi breaks the consumer-software mindset

Consumer question

“Does it have my feature?”

  • their plan mode
  • their permission UX
  • their auth assumptions
  • their workflow opinions
Harness question

“How should this work here?”

  • compose the tools
  • inject the context
  • build the missing extension
  • make the workflow yours
The philosophical move is important. Pi is not trying to convince you to accept a vendor’s worldview. It invites you to treat agent behavior as something you can design, inspect, and change for your environment.

A customization ladder

1

Text files

AGENTS.md, SYSTEM.md, prompts, settings.

2

Skills

Reusable instructions for focused capabilities and workflows.

3

Extensions

Tools, commands, UI, state, event hooks, custom providers.

4

SDK / RPC

Embed or drive the harness from your own application.

Not every customization requires writing TypeScript. Pi gives a ladder: start with files, move to skills, then extensions, then full embedding when you need it.

And yes, you can share it

pi install npm:@your-org/pi-team-tools
pi install git:github.com/your-org/pi-workflows
pi list
Pi packages

Ship prompts, skills, extensions, themes

Team conventions can be versioned, installed, reviewed, and evolved like any other dependency.

This is the practical endpoint: if a workflow matters, don’t keep it in a screenshot or a Slack thread. Package it. Install it. Update it. Review it like software.

Pi is not trying to be the default for everyone

Many agent tools

Opinionated tools for broad audiences

That can be great. Strong defaults, polished workflows, faster onboarding.

Pi’s bet

Specialize the harness to the work

Per repo. Per team. Per model. Per workflow. Less “accept the featureset,” more “shape the agent around the environment.”

This is not an attack on other harnesses. Many are polished, opinionated products for broad audiences. Pi is making a different bet: that advanced users and teams will want to specialize the harness around their environment instead of conforming to one vendor workflow.

Setup

Install

brew install pi-coding-agent

Then run pi.

OAuth

/login

Sign into Claude, ChatGPT Codex, Copilot, Gemini CLI, or Antigravity from interactive mode.

Credential store

~/.pi/agent/auth.json

Stores API keys and OAuth tokens. /logout clears them. OAuth tokens auto-refresh.

Other options

Env vars or custom auth path

ANTHROPIC_API_KEY, OPENAI_API_KEY, or AuthStorage.create("/tmp/my-app/auth.json").

Resolution order: --api-key → auth.json → environment variable.

End with setup right before links: install Pi, run it, authenticate with /login or API keys, and know where credentials live. Auth is flexible too, including custom auth storage paths for embedded SDK use.
